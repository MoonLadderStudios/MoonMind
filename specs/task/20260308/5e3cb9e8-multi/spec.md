## User Scenarios & Testing *(mandatory)*

### User Story 1 - Developer runs local environment (Priority: P1)

Developers need to be able to start the entire MoonMind stack with Temporal using simple commands, so they can develop and test features locally without complex manual setup steps.

**Why this priority**: Without a functional local environment, developers cannot confidently iterate on Temporal-based features.

**Independent Test**: Can run docker compose commands to start the stack, and verify that the Temporal server, temporal-ui, and all necessary worker fleets start and enter a polling state.

**Acceptance Scenarios**:

1. **Given** a fresh clone of the repository, **When** the developer runs the documented `docker compose up` command, **Then** the Temporal server and workers start automatically.
2. **Given** a running local environment, **When** the developer accesses the Temporal UI, **Then** they can see the 'moonmind' namespace and active worker task queues.

---

### User Story 2 - End-to-end task execution (Priority: P1)

An automated test script must exist that validates the entire lifecycle of a task through Temporal, ensuring regressions can be caught automatically during CI or local testing.

**Why this priority**: Validates that the core path of creating and executing a task via Temporal is functional end-to-end.

**Independent Test**: Running the E2E test script creates a task, waits for workers to execute it, and checks that artifacts and final status are correct.

**Acceptance Scenarios**:

1. **Given** a running local environment, **When** the E2E test script is executed, **Then** it submits a task via the API, waits for completion, and asserts that the final state is 'success'.
2. **Given** a successful task run, **When** the E2E test script checks the artifact store and UI endpoints, **Then** it verifies that the expected outputs are present and correctly mapped.

---

### User Story 3 - Environment teardown and clean state (Priority: P2)

Developers need to be able to reset their environment to a clean state easily, and verify that rollback paths work if Temporal services fail or need to be bypassed.

**Why this priority**: Ensures developers can reliably reset state for fresh tests and understand the fallback mechanisms.

**Independent Test**: Can stop the environment, clean volumes, restart, or simulate a rollback to legacy execution.

**Acceptance Scenarios**:

1. **Given** a running environment with task data, **When** the developer runs the cleanup commands, **Then** all execution history and database states are reset.

### Edge Cases

- What happens when a worker container fails to start or crashes?
- How does the system handle missing artifact storage credentials or misconfigured Temporal server connections locally?
- What happens if the namespace initialization job fails or is delayed?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST include a Docker Compose configuration that launches the Temporal server, database, namespace initialization, and all worker fleets (workflow, artifacts, LLM, sandbox, integrations) automatically. (Maps to DOC-REQ-001)
- **FR-002**: System MUST configure worker containers to actively poll their respective task queues upon startup, rather than idling. (Maps to DOC-REQ-001)
- **FR-003**: System MUST provide an automated E2E test script that submits a task, monitors its progress through Temporal workflows, and verifies completion. (Maps to DOC-REQ-002)
- **FR-004**: System MUST verify in the E2E test that artifacts generated by the workflow are accessible via the relevant API. (Maps to DOC-REQ-002)
- **FR-005**: System MUST verify in the E2E test that the task dashboard endpoints return the correct final status matching Temporal's state. (Maps to DOC-REQ-002)
- **FR-006**: System MUST include documentation detailing the local bring-up commands, teardown procedures, and how to execute the E2E test. (Maps to DOC-REQ-001, DOC-REQ-003)
- **FR-007**: System MUST provide a mechanism to test or document the rollback strategy from Temporal to the legacy execution mode. (Maps to DOC-REQ-003)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A developer can successfully bring up the complete Temporal-backed environment locally with a maximum of two commands.
- **SC-002**: The automated E2E test script passes reliably (100% success rate on clean environments) against the local setup.
- **SC-003**: The E2E test completes its validation of task submission, execution, and artifact verification in under 5 minutes.
