# Contract: OAuth Terminal Finalization

## Scope

This contract defines the shared terminal-page and Settings behavior for finalizing an existing OAuth Session into a Provider Profile. It does not define auth-runner startup, PTY bridge internals, or managed-session launch.

## API Contract

### Get OAuth Session

`GET /api/v1/oauth-sessions/{session_id}`

Returns a safe session projection for the authenticated session owner.

Required response fields:

- `session_id`
- `runtime_id`
- `profile_id`
- `status`
- `expires_at`
- `terminal_session_id`
- `terminal_bridge_id`
- `session_transport`
- `failure_reason`
- `created_at`
- `profile_summary`

Response constraints:

- `failure_reason` is sanitized and bounded.
- `profile_summary` is safe for browser display.
- Credential files, token values, raw auth-volume listings, and environment dumps are never returned.

### Attach OAuth Terminal

`POST /api/v1/oauth-sessions/{session_id}/terminal/attach`

Allowed only when:

- the authenticated actor owns the session,
- session status is terminal-attachable,
- the session is not expired,
- terminal refs are present.

Returns one-time attach metadata:

- `session_id`
- `terminal_session_id`
- `terminal_bridge_id`
- `websocket_url`
- `attach_token`
- `expires_at`

### Finalize OAuth Session

`POST /api/v1/oauth-sessions/{session_id}/finalize`

Request body:

- No mutable profile identity or credential-reference fields are accepted.
- Any supplied request body or query values must not override session-owned `profile_id`, `runtime_id`, `volume_ref`, `volume_mount_path`, provider identity, account label, or slot policy.

Eligible states:

- `awaiting_user`
- `verifying`
- `registering_profile`
- `succeeded`

Behavior:

1. For `awaiting_user`, transition to `verifying` and verify durable auth material.
2. On verification success, transition to `registering_profile`.
3. Register or update the Provider Profile selected by the OAuth Session.
4. Transition to `succeeded`.
5. Stop the auth runner if present.
6. Signal the OAuth Session workflow that API finalization succeeded.
7. Return the safe OAuth Session projection with `profile_summary` when available.

Idempotency:

- Repeated finalize requests for `verifying`, `registering_profile`, or `succeeded` converge on the same OAuth Session and Provider Profile.
- No duplicate Provider Profile is created.
- No different `profile_id` is mutated.

Failure:

- Unauthorized sessions return not found or forbidden before verification.
- Cancelled, expired, failed, or superseded sessions fail safely without Provider Profile mutation.
- Verification failures transition to `failed`, stop the auth runner if present, signal workflow failure, and return a sanitized failure reason.

### Cancel OAuth Session

`POST /api/v1/oauth-sessions/{session_id}/cancel`

Allowed for cancellable active states. The terminal page may show this action only when the latest session projection says cancellation is valid or when the status belongs to the known cancellable set.

### Reconnect OAuth Session

`POST /api/v1/oauth-sessions/{session_id}/reconnect`

Allowed for failed, cancelled, or expired sessions. Returns a new safe OAuth Session projection and lets the terminal page navigate or relaunch to the new terminal session.

## Terminal UI Contract

The provider terminal page renders:

- selected profile label or profile ID,
- runtime,
- provider,
- session status,
- expiry,
- sanitized failure or success summary,
- terminal surface when attachable,
- `Finalize Provider Profile` when finalization is eligible,
- Cancel, Retry, or Reconnect only when allowed by state,
- safe registered-profile summary on success,
- optional Return to Settings or manage-profile action after success.

The terminal page calls the same finalize endpoint Settings uses.

The terminal page must not render editable controls for:

- `profile_id`
- `volume_ref`
- `volume_mount_path`
- runtime identity
- provider identity

## Settings Refresh Contract

When finalization succeeds from either Settings or the terminal page:

- Provider Profile query data is invalidated, refreshed, or notified through an equivalent same-browser mechanism.
- Settings may continue to offer its own finalize action for operators who stay there.
- Settings is not required as a post-terminal completion step.

## Security Contract

Browser-visible fields may include:

- OAuth session status,
- terminal I/O generated by provider login CLI,
- timestamps,
- sanitized failure reason,
- registered profile summary.

Browser-visible fields must not include:

- credential files,
- token values,
- raw auth-volume listings,
- environment dumps,
- raw credential contents.
