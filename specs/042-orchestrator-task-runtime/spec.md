## User Scenarios & Testing *(mandatory)*

### User Story 1 - Operate from One Task Surface (Priority: P1)

As an operator, I need orchestrator and queue work to appear in one task list/detail experience so I can monitor and triage all work from a single place.

**Why this priority**: Operational visibility and response speed depend on eliminating split surfaces and inconsistent task terminology.

**Independent Test**: Load the unified list and detail routes with mixed queue/orchestrator data and verify orchestrator records show task-oriented labels and can be inspected without visiting legacy orchestrator pages.

**Acceptance Scenarios**:

1. **Given** queue and orchestrator workloads exist, **When** I open `/tasks/list`, **Then** both appear with normalized runtime/status/title data in one table.
2. **Given** I open a task detail by id, **When** the id belongs to an orchestrator workload, **Then** the shared detail shell loads orchestrator timeline/artifacts without a separate orchestrator-specific route.
3. **Given** legacy bookmarks to `/tasks/orchestrator` or `/tasks/queue`, **When** I navigate to them, **Then** I am redirected to equivalent unified `/tasks/list` filters.

---

### User Story 2 - Author Orchestrator Tasks with Steps and Skills (Priority: P2)

As a task author, I need orchestrator submissions to support explicit ordered steps and skill selections so orchestrator execution can be designed like queue tasks while retaining orchestrator controls.

**Why this priority**: Authoring parity is necessary for predictable orchestration workflows and reduces cognitive switching between runtime modes.

**Independent Test**: Create orchestrator tasks through submit UI and API in both modes (with and without explicit steps) and verify fields, skill catalogs, and execution plan behavior align with the selected authoring method.

**Acceptance Scenarios**:

1. **Given** runtime is set to orchestrator, **When** I open submit form, **Then** I can define ordered steps with skill ids/args and still set target service, priority, and optional approval token.
2. **Given** `steps[]` is omitted, **When** orchestrator task is created, **Then** legacy autogenerated plan behavior remains available for backward compatibility.
3. **Given** `steps[]` is supplied, **When** the task executes, **Then** step execution follows the supplied order with per-step status tracking.

---

### User Story 3 - Finish Tasks During Mid-Run DB Outages (Priority: P3)

As an on-call engineer, I need in-flight orchestrator tasks to keep running when Postgres fails mid-execution so critical operations can complete and state can reconcile later.

**Why this priority**: Resilience during infrastructure incidents is a primary differentiator and avoids partial operations that require manual replay.

**Independent Test**: Start an orchestrator task, simulate Postgres outage mid-run, verify execution continues with artifact-backed state snapshots, then restore DB and confirm reconciliation updates persisted task/step state.

**Acceptance Scenarios**:

1. **Given** Postgres becomes unavailable during step execution, **When** the worker attempts state persistence, **Then** execution continues and state snapshots are written to artifacts instead of aborting the run.
2. **Given** DB connectivity returns after outage, **When** reconciliation runs, **Then** missing task/step state is imported from artifact snapshots and terminal status is synchronized.
3. **Given** queue heartbeat calls fail during outage, **When** execution completes locally, **Then** the worker performs best-effort terminal update once queue API is reachable again.

### Edge Cases

- Both queue and orchestrator sources can plausibly contain the same identifier; detail lookup must resolve deterministically in this order: explicit `source` hint, queue lookup, orchestrator lookup, then a terminal not-found response.
- Explicit step payload includes duplicate or empty step ids; submission validation must reject invalid plans before enqueueing.
- Migration window requests depend on legacy `runId` fields; responses must include both identifiers during transition without contradicting status/state data.
- Postgres flaps repeatedly during one execution; state sink behavior must avoid duplicate or conflicting terminal states after recovery.
- Approval-gated orchestrator tasks submitted through alias endpoints must enforce the same token expiry and policy checks as legacy endpoints.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: User-facing orchestrator terminology MUST migrate from runs to tasks across list/detail/labels while preserving transitional compatibility data where needed. (Maps: DOC-REQ-001, DOC-REQ-005)
- **FR-002**: The platform MUST provide `/orchestrator/tasks*` aliases that mirror legacy `/orchestrator/runs*` behavior, including backward-compatible payload fields (`taskId` and `runId`) during migration. (Maps: DOC-REQ-005)
- **FR-003**: Task dashboard navigation MUST consolidate to `/tasks/list` and `/tasks/:taskId`, with redirects from `/tasks/orchestrator*` and `/tasks/queue*` to equivalent unified destinations. (Maps: DOC-REQ-003, DOC-REQ-006)
- **FR-004**: Unified task list responses MUST normalize queue and orchestrator records into one shared row contract with source, runtime, status, title, and timestamp metadata. (Maps: DOC-REQ-003, DOC-REQ-007)
- **FR-005**: Unified task detail behavior MUST resolve source-specific data for queue and orchestrator ids inside one shared detail shell while preserving timeline/artifact/control visibility for each source, using deterministic lookup order (`source` hint, queue probe, orchestrator probe). (Maps: DOC-REQ-003, DOC-REQ-008)
- **FR-006**: Runtime=orchestrator submit experiences MUST support ordered step authoring with per-step skill ids/args plus orchestrator control fields, and MUST not remove queue-capability fields solely due to orchestrator runtime selection. (Maps: DOC-REQ-002, DOC-REQ-009)
- **FR-007**: Skill discovery APIs MUST expose separate worker and orchestrator skill groups so clients can render runtime-aware skill choices. (Maps: DOC-REQ-010)
- **FR-008**: Orchestrator create requests MUST accept optional `steps[]`; omitted steps MUST preserve legacy autogenerated plans, while supplied steps MUST execute in provided order. (Maps: DOC-REQ-002, DOC-REQ-011)
- **FR-009**: Orchestrator persistence MUST support arbitrary step counts with stable per-step identifiers, order, status, attempts, timestamps, and artifact references independent of fixed legacy plan-step enums. (Maps: DOC-REQ-012)
- **FR-010**: Database migrations MUST transition canonical storage from legacy run tables to task/step tables, migrate existing data, and retire obsolete run tables per agreed naming strategy. (Maps: DOC-REQ-013)
- **FR-011**: Queue dispatch and orchestrator workers MUST support new `orchestrator_task` job payloads while continuing to process legacy `orchestrator_run` jobs without regressions. (Maps: DOC-REQ-014)
- **FR-012**: Execution state recording MUST use a state-sink abstraction that prefers DB persistence, falls back to artifact snapshots on DB failures, and supports reconciliation of missing DB state after recovery. (Maps: DOC-REQ-004, DOC-REQ-015)
- **FR-013**: During DB-related queue heartbeat/lease failures, orchestrator workers MUST continue in-flight execution best-effort and attempt terminal queue-state reconciliation when connectivity returns. (Maps: DOC-REQ-004, DOC-REQ-016)
- **FR-014**: Security controls MUST preserve approval-token enforcement, reject arbitrary-command style skill arguments, and align orchestrator route authentication with existing dashboard task auth expectations. (Maps: DOC-REQ-017)
- **FR-015**: Delivery MUST include production runtime code changes implementing the orchestrator task runtime upgrade; documentation/spec-only outputs are not acceptable for completion. (Maps: DOC-REQ-019)
- **FR-016**: Delivery MUST include validation tests that cover unified dashboard behavior, alias API contracts, worker job-type compatibility, and degraded-mode state sink/reconciliation behavior. (Maps: DOC-REQ-018, DOC-REQ-019)

### Key Entities *(include if feature involves data)*

- **OrchestratorTask**: Canonical orchestrator workload record with task identity, lifecycle status, approval state, runtime metadata, and artifact root.
- **OrchestratorTaskStep**: Ordered step definition and runtime state entry including stable step id, instructions, skill selection, status, attempts, and artifact references.
- **UnifiedTaskRow**: Shared list-view representation used to render queue and orchestrator items in one table with consistent status and runtime semantics.
- **OrchestratorStateSnapshot**: Artifact-backed state capture used when DB writes fail, later consumed by reconciliation to restore missing DB records.
- **OrchestratorTaskDispatch**: Queue payload envelope that links a task id and execution metadata to worker-side processing for new and legacy job types.

### Assumptions & Dependencies

- Agent Queue remains DB-backed for claiming new work; this feature focuses on resilience for already-claimed in-flight orchestrator execution.
- Existing dashboard architecture remains in the current thin-dashboard stack; no framework rewrite is required.
- Existing approval policy and skill-execution policy helpers remain the source of truth for authorization and guardrails.
- Artifact storage paths for orchestrator runs/tasks remain durable enough to act as fallback state snapshots during DB outages.

### Non-Goals

- Replacing the entire Agent Queue persistence model in this iteration.
- Building a generalized workflow engine that unifies every workload into one table immediately.
- Replatforming the dashboard to a different frontend framework.

- **SC-005**: Worker compatibility tests pass for both `orchestrator_task` and `orchestrator_run` queue job types with no regression in step event emission.
- **SC-006**: Security and auth regression tests confirm approval enforcement, skill-arg command restrictions, and authenticated access parity for orchestrator task routes.
