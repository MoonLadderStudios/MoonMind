# Data Model: Claude Browser Terminal Sign-In Ceremony

## OAuth Session

- `session_id`: Stable OAuth session identifier.
- `runtime_id`: Runtime identifier; MM-479 requires `claude_code` coverage.
- `profile_id`: Provider profile row, normally `claude_anthropic`.
- `status`: Session lifecycle state. Terminal attach is allowed for `bridge_ready`, `awaiting_user`, and `verifying`.
- `terminal_session_id`: Terminal session identifier exposed after the auth-runner bridge is ready.
- `terminal_bridge_id`: PTY/WebSocket bridge identifier exposed after bridge startup.
- `requested_by_user_id`: Owner used to scope API access.
- `expires_at`: Session expiration; attach tokens are bounded by this lifetime.
- `metadata_json`: Secret-free session metadata. For attach, stores token hash, used flag, issued timestamp, terminal connection timestamps, close reason, and bounded terminal counters.

## OAuth Terminal Attach Token

- Generated by `POST /api/v1/oauth-sessions/{session_id}/terminal/attach`.
- Returned once to the browser as `attach_token` and embedded in `websocket_url`.
- Stored server-side only as `terminal_attach_token_sha256`.
- Consumed by `/{session_id}/terminal/ws` and marked used before PTY connection starts.
- Invalid when session is missing, owned by another user at attach time, expired, not terminal-ready, missing terminal bridge identifiers, already used, or hash mismatch.

## Terminal Bridge Connection

- Tracks session/bridge ownership metadata.
- Accepts input, resize, heartbeat, and close frames.
- Rejects generic Docker/task execution frames.
- Forwards input bytes to the auth-runner PTY.
- Exposes only secret-free metadata: input/output counts, heartbeat count, and last resize dimensions.

## Claude Auth Runner PTY

- Short-lived PTY connected to the Claude OAuth auth-runner.
- Receives exact pasted token or authorization-code input.
- Writes durable credential output only into the mounted Claude home volume, outside this story's data model.

## State Transitions

```text
pending/starting -> bridge_ready -> awaiting_user -> verifying -> registering_profile -> succeeded
                                      |              |                         |
                                      v              v                         v
                                  cancelled       failed                    failed
```

MM-479 relies on `awaiting_user` remaining terminal-attachable until the operator pastes the returned Claude token or code.
