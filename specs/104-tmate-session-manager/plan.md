# Implementation Plan: TmateSessionManager

**Branch**: `104-tmate-session-manager` | **Date**: 2026-03-24 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/104-tmate-session-manager/spec.md`

## Summary

Extract the tmate session lifecycle into a shared `TmateSessionManager` class (`moonmind/workflows/temporal/runtime/tmate_session.py`), refactor both `ManagedRuntimeLauncher.launch()` and `oauth_session_activities.start_auth_runner()` to delegate to it, and wire self-hosted server configuration via `TmateServerConfig`. This unifies the tmate strategy across runtime wrapping and OAuth sessions, eliminating the current split where two code paths independently manage sockets, configs, readiness detection, and endpoint extraction.

## Implementation Phases

### Phase 1: TmateSessionManager Core

1. Create `moonmind/workflows/temporal/runtime/tmate_session.py`:
   - `TmateServerConfig` dataclass (host, port, fingerprints)
   - `TmateEndpoints` dataclass (session_name, socket_path, attach_ro, attach_rw, web_ro, web_rw)
   - `TmateSessionManager` class:
     - `__init__(session_name, socket_dir, server_config)` 
     - `is_available()` — `shutil.which("tmate")` check
     - `start(command, env, cwd, exit_code_capture, timeout_seconds)` — create socket dir, write config, launch subprocess, wait for readiness, extract endpoints
     - `endpoints` property
     - `exit_code_path` property
     - `teardown()` — kill process, cleanup files

2. Wire `TmateServerConfig` from environment variables in settings or a factory function `TmateServerConfig.from_env()`.

3. Port the config file generation: write `TMATE_FOREGROUND_RESTART_OFF` plus optional server `set-option` directives.

4. Port the endpoint extraction: `tmate -S <sock> display -p '#{<key>}'` for all 4 endpoint types.

5. Port the subprocess launch: `tmate -S <sock> -f <conf> -F new-session -A -s <session> <command>`.

### Phase 2: Launcher Refactor

1. Replace inline tmate logic (launcher.py lines 426-527) with `TmateSessionManager` delegation:
   - `if TmateSessionManager.is_available():` → create manager, call `start()`, build endpoints dict from `TmateEndpoints`
   - Preserve the existing endpoints dict structure (`tmate_session_name`, `tmate_socket_path`, etc.) for supervisor compatibility
   - Keep the exit code capture env var (`MM_EXIT_FILE`)
   - Preserve the headless fallback when tmate is unavailable

2. Call `teardown()` is NOT done in the launcher — it's done by the supervisor. The launcher returns the manager instance (or its endpoints) so the supervisor can call teardown later.

### Phase 3: OAuth Activities Refactor

1. Refactor `oauth_session_activities.start_auth_runner()`:
   - The tmate runs inside a Docker container, so `TmateSessionManager` isn't directly used as a subprocess manager inside the activity. Instead:
   - Replace the hardcoded Docker-exec polling loop with a helper that uses the same endpoint extraction command format as `TmateSessionManager`
   - Use the `TmateSessionManager` endpoint key constants for consistency
   - This ensures both consumers use the same tmate command patterns even though the OAuth case goes through Docker exec

### Phase 4: Tests

1. Create `tests/unit/workflows/temporal/runtime/test_tmate_session.py`:
   - Test `is_available()` with and without tmate on PATH
   - Test `TmateServerConfig.from_env()` with various env combinations
   - Test config file generation with and without server config
   - Test `TmateEndpoints` construction
   - Test `teardown()` cleanup logic
   - Test `start()` error handling (timeout, subprocess failure)

2. Update existing launcher tests to verify delegation behavior
3. Update existing OAuth activity tests if any mock the tmate polling loop
